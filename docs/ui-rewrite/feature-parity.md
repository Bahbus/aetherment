# Aetherment Plugin UI Rewrite – Feature Parity Inventory

This document enumerates every currently user-visible plugin feature with stable IDs for migration tracking.

| Feature ID | Feature | Criticality | Current behavior summary | Trigger path in existing UI | Source code location (file + class/function) | Data/config dependencies | Persistence or IO side effects | Error states and user feedback behavior | Desktop-sync touchpoints |
|---|---|---|---|---|---|---|---|---|---|
| AETH-UI-001 | Active preset detection and label resolution | important | The preset selector resolves whether current settings match Default, Custom, built-in presets, or saved user presets and displays that effective label. | Mods -> Mod detail page -> Preset combo label. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage` (preset resolution flow). | `mod_manager.metas`, `mod_manager.settings`, active collection settings map. | None (read-only calculation). | No explicit error UI; fallback behavior displays Custom/Default when no exact preset match. | Indirect: resolved state reflects settings synchronized through Penumbra-backed backend.
| AETH-UI-002 | Apply built-in preset | critical | Selecting a built-in preset writes all mapped option values into the active collection settings for the selected mod. | Mods -> Mod detail page -> Preset combo -> built-in preset item click. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage` (`set_settings` closure path). | Built-in preset definitions from mod meta, selected mod ID, selected collection settings map. | Mutates in-memory settings; change is flushed to settings persistence and backend apply pipeline. | No hard failure prompt in this interaction; downstream apply/persist errors surface through global progress/messages. | Yes: updates pass through backend/Penumbra sync paths when apply occurs.
| AETH-UI-003 | Apply user preset | critical | Selecting a user preset applies its saved option values to current mod settings in active collection. | Mods -> Mod detail page -> Preset combo -> user preset item click. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`. | User preset list from `mod_settings.presets`, active settings map. | Mutates settings; persisted/applied in standard save/apply flow. | Invalid/missing preset index is naturally prevented by UI list generation; downstream failures use global progress/errors. | Yes: applied values propagate to Penumbra via backend apply.
| AETH-UI-004 | Create/overwrite user preset | important | User can name a preset and save current option state; if name exists, behavior is overwrite/update of stored preset data. | Mods -> Mod detail page -> preset editor row (`+` and name field). | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`. | `new_preset_name`, active option settings snapshot, per-mod preset store. | Writes per-mod preset data via `mod_settings.save(mod_id)`. | Minimal inline validation; empty/invalid names are effectively non-action or non-useful entries depending input state; no dedicated modal errors. | None direct; local preset persistence only.
| AETH-UI-005 | Delete user preset | important | Removes a selected user preset from the saved preset list for the mod. | Mods -> Mod detail page -> preset popup list -> delete icon/button. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`. | User preset vector/index state. | Updates preset list and saves via `mod_settings.save(mod_id)`. | No confirmation dialog in current path; user sees immediate list update. | None direct.
| AETH-UI-006 | Export preset to clipboard | nice-to-have | Serializes a preset as JSON, base64-encodes it, and writes to system clipboard for sharing/backup. | Mods -> Mod detail page -> preset popup list -> clipboard icon/button. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`. | Selected preset serializable structure, clipboard service from UI runtime. | Clipboard write (OS IO boundary). | Clipboard/serialization failures are logged or silently fail depending platform hook behavior; no dedicated rich UI error modal. | None direct.
| AETH-UI-007 | Import preset from clipboard | important | Reads clipboard text, decodes base64 JSON into `Preset`, and adds/replaces preset entry. | Mods -> Mod detail page -> "Import preset from clipboard" button. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`. | Clipboard text, JSON/base64 decode path, preset storage. | Mutates preset store and persists with `mod_settings.save(mod_id)`. | Invalid payload paths log an error and avoid destructive writes; user feedback is primarily non-success (no preset added) plus log/progress surfaces. | None direct.
| AETH-UI-008 | Import `.aeth` mods from file dialog | critical | Opens OS file picker (multi-select), filters `.aeth`, and starts async install tasks for selected files. | Mods -> left panel -> "Import Mods" button. | `aetherment/src/view/mods.rs` -> `Mods::draw_modlist_simple`. | `config.file_dialog_path`, selected files, task/progress handles, install backend routines. | File IO reads package files; updates remembered dialog path; installs/unpacks mod content. | Install/import failures surface through task progress + error message stack in core UI. | Yes: imported mods become visible/applicable via backend/Penumbra-managed state.
| AETH-UI-009 | Enable/disable mod for active collection | critical | Checkbox toggles whether selected mod is enabled in currently selected collection. | Mods -> Mod detail page -> "Enabled" checkbox. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`; backend bridge in `aetherment/src/modman/backend/penumbra_ipc.rs`. | Selected collection ID, mod ID, backend connection state. | IPC/backend mutation of mod-enabled state; queued apply/update effects. | Backend errors propagate via global notification/progress paths rather than inline checkbox error UI. | Yes: direct Penumbra setting mutation and sync.
| AETH-UI-010 | Auto-update toggle for remote-backed mods | important | Shows and toggles auto-update only when mod has remote origin metadata; persists user preference. | Mods -> Mod detail page -> "Auto Update" checkbox (conditional). | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`; remote settings in `aetherment/src/remote/settings.rs`. | `remote_settings.origin`, `remote_settings.auto_update`, mod ID. | Persists remote settings via `remote_settings.save(mod_id)`. | Save/persistence failures are surfaced via shared error infrastructure/logging; no dedicated inline retry UI. | Potential indirect sync when update checks/install flows run.
| AETH-UI-011 | Option editor (single/multi/group/path/slider/color) | critical | Dynamic schema-driven option controls allow changing mod variant choices and parameters across multiple value types. | Mods -> Mod detail page -> Options area (and category tabs). | `aetherment/src/view/mods.rs` -> `draw_option`, `Mods::draw_modpage`. | Meta option schema, current settings map, category selection state. | Mutates settings map; changes eventually persisted and applied through backend flow. | Invalid values are generally constrained by control widgets; backend/apply errors appear in global status messages. | Yes: resulting settings synchronize through backend/Penumbra apply.
| AETH-UI-012 | Category tab filtering for options | nice-to-have | Category tabs filter visible options without changing data until user edits controls. | Mods -> Mod detail page -> category tab row. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`. | Option category metadata, `selected_category_tab`. | None direct. | No explicit errors; if category missing/empty, corresponding option area shows less/no items. | None.
| AETH-UI-013 | Collection selector for editing target | critical | Dropdown switches which Penumbra collection’s settings are being edited by the UI. | Mods -> left panel -> Collection combo. | `aetherment/src/view/mods.rs` -> `Mods::draw_modlist_simple`. | `mod_manager.collections`, `config.active_collection`. | Persists `config.active_collection` to config JSON. | Invalid collection scenarios are mitigated by list-driven selection; fallback selection handled at load time. | Yes: determines which synced Penumbra collection receives edits.
| AETH-UI-014 | Select mod from list and open detail pane | critical | Clicking mod in list selects it and loads its options/details for editing. | Mods -> left panel -> mod list item click. | `aetherment/src/view/mods.rs` -> `Mods::draw_modlist_simple`. | Sorted mod list, metadata maps, selected mod state, asset loader cache. | Clears cached `aetherment://` textures for refresh behavior. | Missing metadata/resource cases fall back to reduced rendering and/or downstream resource error displays. | Indirect via showing synced state for selected mod.
| AETH-UI-015 | Reload mods index | important | Manual refresh reloads mod manager state and rediscovery of mods. | Mods -> left panel -> "Reload Mods" button. | `aetherment/src/view/mods.rs` -> `Mods::draw_modlist_simple`; manager logic in `aetherment/src/modman/manager.rs`. | Backend idle/busy state, mod manager instance. | Triggers reload operations (filesystem/backend queries). | Reload failures bubble into progress/messages logs; button may be unavailable while busy. | Yes: refreshes synced mod/collection view from backend sources.
| AETH-UI-016 | Manual apply queued changes | critical | User can force immediate apply of queued setting changes rather than waiting for auto apply. | Mods -> left panel -> "Manually Apply Now" button (when queue non-empty). | `aetherment/src/view/mods.rs` -> `Mods::draw_modlist_simple`, `Mods::check_apply`. | Apply queue size, task executor/progress state, backend connection. | Spawns background apply/finalize work; may trigger redraw side effects. | Applying/success/failure status communicated via notifications and progress/message UI. | Yes: core synchronization action to Penumbra/runtime state.
| AETH-UI-017 | Global auto-apply enable/disable | important | Controls whether setting changes auto-apply in background without manual button press. | Settings view -> "Auto apply" checkbox. | `aetherment/src/view/settings.rs` -> `Settings::ui`; usage in `aetherment/src/view/mods.rs` -> `Mods::check_apply`. | `config.auto_apply_enabled`. | Persists config JSON. | Misconfiguration is low-risk; behavior differences surfaced by queue/apply status notifications. | Yes: governs cadence of desktop-side sync/apply.
| AETH-UI-018 | Auto-apply redraw self option | nice-to-have | Optional redraw after successful apply to refresh character appearance. | Settings view -> "Redraw self" checkbox. | `aetherment/src/view/settings.rs` -> `Settings::ui`; redraw call path in `aetherment/src/view/mods.rs` -> `Mods::check_apply`. | `config.auto_apply_redraw`, backend redraw capability. | Persists config and may invoke `backend.redraw_self()` after apply. | Redraw failure/non-availability is non-fatal and surfaced through backend/log channels. | Yes: direct runtime sync/refresh touchpoint.
| AETH-UI-019 | Open plugin window on launch | nice-to-have | Startup preference decides whether the plugin UI opens automatically at plugin initialization. | Settings view checkbox + plugin startup behavior. | `aetherment/src/view/settings.rs` -> `Settings::ui`; bridge `plugin/plugin/Native.cs` -> `config_plugin_open_on_launch`; startup read in `plugin/plugin/Aetherment.cs`. | `config.plugin_open_on_launch`, native state handle. | Persists config; startup behavior IO through native bridge. | No dedicated runtime error dialog; failure means window does not auto-open as expected. | Yes: startup handshake between desktop plugin host and native core.
| AETH-UI-020 | Apply Dalamud style from mod profile | nice-to-have | Button applies mod-provided Dalamud style configuration with gamma parameterization. | Mods -> Mod detail page -> "Set Dalamud Style" button + gamma slider. | `aetherment/src/view/mods.rs` -> `Mods::draw_modpage`; related plugin service in `plugin/plugin/Service/UiColor.cs`. | `meta.plugin_settings.dalamud`, gamma value, optional service availability. | Runtime style/hook updates to UI color tables (when integration available). | If integration unavailable, action is absent/no-op depending state; no hard-fail modal. | Yes: touches plugin-host UI color hook and native bridge.
| AETH-UI-021 | Progress bars for background tasks | important | Top-level progress bars display main/subtask completion and status text during long operations. | Global/top-level core draw area. | `aetherment/src/lib.rs` -> `Core::draw`. | `TaskProgress` state (main/sub progress/messages). | None direct (display only). | Conveys ongoing operation state; complements error message stack for failures. | Indirect: reflects sync/import/apply operations.
| AETH-UI-022 | Completion/error message stack with acknowledge | important | Message queue displays success/error lines and allows clearing via `Ok`. | Global/top-level core draw area -> message list and `Ok` button. | `aetherment/src/lib.rs` -> `Core::draw`. | `TaskProgress` message queue fields. | Acknowledge clears/reset progress/message state in memory. | Primary user feedback channel for async errors and completion notices. | Indirect: reports sync/desktop operation outcomes.
| AETH-UI-023 | Auto-apply lifecycle notifications | important | Notification text cycles through applying/success/error states for auto-apply workflow. | Triggered during mod/settings interactions that enqueue apply; shown in UI notification area. | `aetherment/src/view/mods.rs` -> `Mods::check_apply`. | Queue size, busy/error state, config auto-apply flags. | Notification state writes; may invoke redraw side effect when enabled. | Explicit user-facing statuses for apply lifecycle, including error wording. | Yes: tightly coupled to apply/sync pipeline.
